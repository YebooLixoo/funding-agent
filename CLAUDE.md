# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## One consolidated system

- **`web/`** — FastAPI app + Click CLI (`web/cli.py`). FastAPI hosts no background work; the CLI is what launchd schedules.
- **`src/`** — fetch/filter/summarize library (no DB coupling). Imported by `web/services/fetch_runner.py`.
- **`data/platform.db`** — single SQLite source of truth (WAL mode).
- **`alembic/`** — authoritative schema; baseline + drift + consolidate revisions land cleanly on fresh installs and the live DB.
- **launchd plists** in `launchd/`: fetch (Thu 12pm), email-digest (hourly), backup (daily 2am).

`web/services/seed_opportunities.py` and `launchd/com.boyu.funding-agent.{daily,weekly}.plist` are gone (replaced by the consolidated pipeline).

## Common commands

```bash
# Install
uv sync                                  # backend deps (uv-managed; pyproject.toml)
cd frontend && npm install               # frontend deps

# Dev (web platform — both servers; SQLite by default)
./scripts/start_platform.sh
# Or run them separately:
uv run uvicorn web.main:app --reload --port 8000
cd frontend && npx vite --port 5173

# Admin CLI (also what launchd schedules)
uv run python -m web.cli fetch                       # fetch pipeline (acquires data/.fetch.lock)
uv run python -m web.cli email-digest --due          # dispatch to all due users (hourly cadence)
uv run python -m web.cli email-digest --user-email me@x --test   # test send to one user
uv run python -m web.cli regenerate-history          # rebuild docs/index.html

# Wrappers:
./scripts/fetch_now.sh                   # = web.cli fetch
./scripts/email_now.sh                   # test mode (requires ADMIN_EMAIL)
./scripts/email_now.sh --prod            # = web.cli email-digest --due

# Tests
uv run python -m pytest tests/ -v
uv run python -m pytest tests/test_filter.py::test_name -v   # single test

# Lint
uv run ruff check .

# Frontend build
cd frontend && npm run build             # tsc -b && vite build
```

## Configuration layering

Hydra-style: `conf/config.yaml` is the root and merges in `conf/sources/*.yaml` (government, industry, university, compute), `conf/filter.yaml`, and `conf/email.yaml`. `weekly_fetch.py` and `weekly_email.py` each call `load_config()` to do the merge manually (not via `@hydra.main`), so adding a new config file means editing the `extra` list in those scripts.

`conf/email.yaml` is gitignored — copy from `conf/email.yaml.example`. Real credentials live in `.env` (`OPENAI_API_KEY`, `GMAIL_ADDRESS`, `GMAIL_APP_PASSWORD`, `DATABASE_URL`, `JWT_SECRET_KEY`).

## Adding a funding source

- **API source** (NSF/NIH/Grants.gov-like): subclass `BaseFetcher` in `src/fetcher/`, register with `@register_fetcher("name")`, import in `src/fetcher/__init__.py`, and wire into `fetch_government()` in `src/weekly_fetch.py`.
- **Scraped web source**: just add an entry under `web_sources` (government), `sources` (industry/university), or the appropriate category list in `conf/sources/compute.yaml`. `WebScraperFetcher` handles them generically.
- **Curated compute sources bypass the keyword/LLM filter** (see `weekly_fetch.py:395`). They're pre-vetted via YAML curation + `OpportunityValidator`. Don't add filtering logic here without understanding why.

## Bootstrap mechanism

New sources get a "bootstrap" first-fetch with no date lower bound (pulls all open opportunities, not just the last 7 days). Tracked in `source_bootstrap` table via `StateDB.seed_bootstrap()` / `mark_source_bootstrapped()`. After the first successful fetch, the source falls back to the normal 7-day window.

## Scoring (web platform)

`web/services/scoring.py` combines four signals into a final per-user score:
- Keyword match (40%) via `src.filter.keyword_filter.KeywordFilter`, weighted by per-keyword `UserKeyword.weight`
- Profile match (30%) — text similarity to user profile
- Behavioral (20%) — boost for bookmarks, penalize from dismissals on similar opps
- Urgency (10%) — deadline proximity + recency

The internal pipeline uses only the keyword + LLM borderline filter (no per-user profile). They produce different scores by design.

## Static history page

`docs/index.html` is a static page of all emailed opportunities, generated by `src/history_generator.py` and published via GitHub Pages. **Always regenerate it after changes that affect emailed-opportunity display** — `weekly_fetch.py` and `weekly_email.py` both call it automatically; if you mutate the DB or the template (`templates/history.html`) by hand, run `HistoryGenerator(...).generate(db)` or re-run the pipeline.

## Database notes

- Single SQLite database at `data/platform.db` (WAL mode). Used by both FastAPI and the CLI.
- **Alembic is authoritative.** `alembic/versions/` contains the baseline + reconcile + consolidate revisions; `Base.metadata.create_all` is no longer called on startup. To change schema: edit the model, then `uv run alembic revision --autogenerate -m "..."`, then `uv run alembic upgrade head`.
- Daily SQLite `.backup` (launchd) writes to `data/backups/platform-YYYYMMDD.db`.
- `data/` is gitignored.

## Time zones and scheduling

The pipeline still assumes **Mountain Time** for fetch-window math (`src/utils.py:last_thursday_noon_mt`, `now_mt`). launchd plists in `launchd/`:

- `com.boyu.funding-agent.fetch.plist` — Thursday 12:00 → `web.cli fetch`
- `com.boyu.funding-agent.email.plist` — hourly (`StartInterval=3600`) → `web.cli email-digest --due` (each user's pref governs whether *they* are due)
- `com.boyu.funding-agent.backup.plist` — daily 02:00 → `sqlite3 data/platform.db .backup ...`

User email-pref `day_of_week` follows **Python `datetime.weekday()`**: 0=Mon..6=Sun, so Thursday = 3 (not 4). See `web/models/email_pref.py`.

## Environment quirks

- Python 3.10+ via `.python-version`; `uv` manages the venv
- `pyproject.toml` lists fewer deps than actually used at runtime (web platform deps like FastAPI, SQLAlchemy, asyncpg are installed via `uv sync` from `uv.lock`)
- Ruff line length is 100, target py310
